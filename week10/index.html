Week 10

MDN: Validating Forms

  • To ensure an application is secure, what types of validation should be considered?

    • An application should always perform security checks on any form-submitted data on the server-side as well as the client-side.

    What are three reasons for validating forms?

    1. We want to get the right data, in the right format.
    2. We want to protect our users' data.
    3. We want to protect ourselves.

    What are the two types of client-side validation?

    • Built-in form validation.

      • required - Specifies whether a form field needs to be filled.
      • minlength and maxlength - Specifies the minimum and maximum length of textual data.
      • min and max - Specifies the minimum and maximum values of numerical input types.
      • types - Specifies whether the data needs to be a number, an email address, or some other specific preset type.
      • pattern - Specifies a regular expression that defines a pattern the entered data needs to follow.
    • JavaScript validation.

      • When would I use JavaScript validation?

        If you want to take control over the look and feel of native error messages or to deal with legacy browsers that do not support HTML's built-in form validation.

      • Constraint Validation API

          DOM interfaces:

          • HTMLButtonElement - Represents a <button> element.
          • HTMLFieldSetElement - Represents a <fieldset> element.
          • HTMLInputElement - Represents an <input> element.
          • HTMLOutputElement - Represents an <output> element.
          • HTMLSelectElement - Represents a <select> element.
          • HTMLTextAreaElement - Represents a <textarea> element.

          Common Properties

          • validationMessage - Returns a localized message describing the validation constraints that the control does not satisfy.
          • validity - Returns a validityState object that contains several properties describing the validitiy state of the element, including:
            • patternMismatch - Returns true if the value does not match the specified pattern.
            • tooLong - Returns true if the value is longer than the maximum length specified by the maxlength attribute.
            • tooShort - Returns true if the value is shorter than the minimu length specified by the minlength attribute.
            • rangeOverflow - Returns true if the value is greater than the maximum specified by the max attribute.
            • rangeUnderflow - Returns true if the valuye is less than the minimum specified by the min attribute.
            • typeMismatch - Returns true if the value is not in the required syntax (when type is email or url).
            • valid - Returns true if the element meets all its validation constraints.
          • willValidate - Returns true if the element will be validated when the form is submitted.

          Common Methods

          • checkValidity() - Returns true if the element's value has no validity problems.
          • reportVlidity() - Reports invalid field(s) using events.
          • setCustomValidity(message) - Adds a custom error message to the element.

MDN: Using the Fetch API

  • What is the Fetch API?

    The Fetch API provides a JavaScript interface for accessing and manipulating parts of the HTTP pipeline, such as requests and responses.

  • Basic Fetch API example

    fetch('http://example.com/movies.json')

    .then(response => response.json())

    .then(data => console.log(data));

  • Properties of fetch config

    • method - HTTP method
    • mode - no-cors, *cors, same-origin
    • cache - default, no-cache, reload, force-cache, only-if-cached
    • credentials - include, *same-origin, omit
    • headers - nested object consisting of HTTP headers ('Content-Type': 'application/json')
    • redirect - manual, *follow, error
    • referrerPolicy - no-referrer, *no-referrer-when-downgrade, origin, origin-when-cross-origin, same-origin, strict-origin, strict-origin-when-cross-origin, unsafe-url
    • body - body data, must match "Content-Type" header (JSON.stringify(data))

  • Classes

    • Request - Used to build the request ahead of time.

      const myRequest = new Request('url', { method: 'GET', ... });

    • Headers - Allows you to create your own headers.

      const myHeaders = new Headers();

      myHeaders.append('Content-Type', 'text/plain');

      Possible guard values:

      • none - default
      • request - guard for a headers object obtained from a request.
      • request-no-cors - guard for a headers object obtained from a request created with Request.mode no-cors.
      • response - guard for a headers object obtained from a response.
      • immutable - guard that renders a headers object read-only; mostly used for ServiceWorkers.

        • Response object properties

        • Response.status - An integer (default value 200) containing the response status code.
        • Response.statusText - A string (default value ""), that corresponds to the HTTP status code message.
        • Response.ok - Shorthand for checking that a status is in the range 200-299 inclusive.

        Body types

        • ArrayBuffer
        • ArrayBufferView
        • Blob/File
        • string
        • URLSearchParams
        • FormData

        Body methods

        • Request.arrayBuffer() / Response.arrayBuffer()
        • Request.blob() / Response.blob()
        • Request.formData() / Response.formData()
        • Request.json() / Response.json()
        • Request.text() / Response.text()
